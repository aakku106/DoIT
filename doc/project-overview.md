# DoIT Project Overview

## What this project is

This is a Go CLI todo app using:

- SQLite for storage
- sqlc for generating typed Go query code from SQL files

The app is intended to run commands like:

- `todo add "task title"`
- `todo list`
- `todo done 3`

## Main architecture

High-level runtime flow:

1. CLI command starts the app.
2. Entrypoint reads command-line args.
3. CLI handlers call generated store/query methods.
4. Store layer runs SQL against SQLite.

Dependency chain:

- `cmd/todo/main.go` -> `internal/cli/commands.go`
- `internal/cli/commands.go` -> `internal/store/*.go` (generated by sqlc)
- `internal/store/*.go` -> SQLite DB connection from `internal/db/sqlite.go`
- SQL source files used to generate store code:
  - `sql/schema.sql`
  - `sql/queries.sql`
  - configured by `sqlc.yml`

## File-by-file purpose

### Entrypoint and command routing

- `cmd/todo/main.go`
  - Starts DB connection.
  - Creates `store.Queries` with `store.New(dbConn)`.
  - Routes CLI args (`add`, `list`, `done`) to functions in `internal/cli`.

### DB bootstrap

- `internal/db/sqlite.go`
  - Opens SQLite DB (`todo.db`).
  - Pings DB.
  - Ensures `todos` table exists via inline schema SQL.

### CLI handlers

- `internal/cli/commands.go`
  - Intended place for business/command logic.
  - Should call generated store methods (`CreateTodo`, `ListTodos`, `CompleteTodo`).
  - Currently incomplete/partially stubbed.

### Generated SQL access layer (sqlc output)

- `internal/store/db.go`
  - Defines query wrapper and DBTX interface.
- `internal/store/models.go`
  - Generated `Todo` model struct.
- `internal/store/querier.go`
  - Generated interface with typed query methods.
- `internal/store/queries.sql.go`
  - Generated implementations for:
    - `CreateTodo(ctx, params)`
    - `ListTodos(ctx, session)`
    - `CompleteTodo(ctx, id)`
    - `DeleteTodo(ctx, id)`

### SQL definitions (source of truth for sqlc)

- `sql/schema.sql`
  - Defines `todos` table schema.
- `sql/queries.sql`
  - Defines named queries used by sqlc.

### Migration files

- `migrations/001_init.sql`
  - Also defines the same `todos` table.
  - Currently duplicates schema found in `sql/schema.sql` and inline schema in DB init.

### Extra/in-progress code

- `internal/todo/service.go`
  - Contains a `Service` struct with `store *store.Queries`.
  - Not yet used by entrypoint/CLI.

### Experimental folders (not main runtime)

- `internal/tempTest/sqlc-tutorial/...`
- `internal/tempTest/try2/...`
  These appear to be sqlc learning experiments/sandboxes and are separate from the main app execution path.

## Current broken links / why app feels disconnected

1. `add` route passes wrong argument in `main.go`

- Current code passes `args[1]` (the word `add`) as title.
- Should pass title text from the next arg(s).

1. `done` route has type mismatch in `main.go`

- `args[2]` is string.
- `CompleteTodo` currently expects `int64`.
- Needs parsing (e.g., `strconv.ParseInt`).

1. `ListTodos` call signature mismatch in `commands.go`

- Generated method requires `(context.Context, session string)`.
- Current call has no arguments.

1. CLI handlers are stubs/incomplete

- `AddTodo` mostly prints debug output.
- `CompleteTodo` is empty.

1. Multiple schema sources

- Inline schema in `internal/db/sqlite.go`.
- Schema in `sql/schema.sql`.
- Migration in `migrations/001_init.sql`.
- This works short-term but creates maintenance confusion.

## How sqlc fits in

`sqlc.yml` points sqlc to schema and query files:

- schema: `sql/schema.sql`
- queries: `sql/queries.sql`
- output: `internal/store`

When you run sqlc generate, it regenerates the store files in `internal/store`.

Rule of thumb:

- Edit SQL files (`sql/*.sql`) for DB/query behavior.
- Regenerate code.
- Do not hand-edit generated files in `internal/store`.
